Git 学习笔记

整理自我第一次把个人主页推上 GitHub 的真实过程。踩过的坑、搞懂的命令，都写在这里。

目录

Git 是什么 第一次配置 本地第一次提交 关联远程仓库 推送到 GitHub push 被拒绝怎么办 换电脑怎么拉代码 日常更新流程 ZIP 和 clone 的区别
Step 0

Git 是什么？

Git 是一个版本控制工具。你可以把它理解成项目的「时光机」： 每次 commit 都是一次快照，随时能回到之前的版本，也能把本地改动同步到 GitHub。

  • 本地仓库：你电脑上的项目 + `.git` 隐藏文件夹（记录历史）
  • 远程仓库：GitHub 上的同名项目（比如 `QIII01.github.io`）
  • 常见命令：`add` 暂存 → `commit` 提交 → `push` 上传
小提示：个人主页用 GitHub Pages 时，仓库名通常是 用户名.github.io，推送成功后网站地址是 https://用户名.github.io。
Step 1

第一次配置身份

Git 需要知道「谁」在提交。每台新电脑只需配置一次：

git config --global user.name "你的名字" git config --global user.email "你的GitHub邮箱" # 检查是否成功 git config --global user.name git config --global user.email

邮箱建议和 GitHub 账号一致，提交记录会正确关联到你的账号。

Step 2

本地第一次提交

如果文件夹里还没有 Git 历史，需要先初始化，再把文件加入暂存区并提交：

cd 你的项目目录 git init git add index.html git status git commit -m "Initial commit: personal homepage" git log --oneline
  • git add：把改动放进「暂存区」，准备提交
  • git commit：在本地生成一条历史记录
  • git log：查看已有 commit；若提示 does not have any commits yet，说明还没 commit 成功
我踩过的坑：运行 git commit 时不加 -m 会打开编辑器。 如果第一行留空，Git 会取消提交。新手建议直接用 git commit -m "说明"。
Step 3

关联远程仓库

本地 commit 完成后，还要告诉 Git：GitHub 上的仓库在哪里。

git remote add origin https://github.com/QIII01/QIII01.github.io.git git remote -v

执行 git remote -v 后如果看到：

origin https://github.com/QIII01/QIII01.github.io.git (fetch) origin https://github.com/QIII01/QIII01.github.io.git (push)
  • origin：远程仓库的别名，默认就叫 origin
  • (fetch)：从这个地址拉取更新（git pull / git fetch）
  • (push)：往这个地址上传提交（git push）

两行地址一样，说明拉和推都指向同一个 GitHub 仓库，配置成功。

Step 4

推送到 GitHub

第一次推送建议带上 -u，以后在同一分支可以直接 git push：

git push -u origin main

成功时终端会出现类似输出：

branch 'main' set up to track 'origin/main'.

这表示本地 main 已经和远程 origin/main 关联好了。

补充：第一次 push 可能需要登录 GitHub。HTTPS 方式下，密码栏通常要填 Personal Access Token（PAT），不是账号登录密码。
Step 5

push 被拒绝怎么办？

如果看到：

! [rejected] main -> main (fetch first) Updates were rejected because the remote contains work that you do not have locally.

意思是：GitHub 上已经有 commit，而你本地是另一套历史（常见于从 GitHub 下载 ZIP 后重新 git init）。Git 不允许直接覆盖，所以 push 失败。

两种处理方式：

  1. 以本地为准（个人项目常用）：确认本地是最新版本后，强制推送：
    git push -u origin main --force
    成功时会看到 (forced update)。这会重写 GitHub 上的历史，协作项目要谨慎。
  2. 保留远程历史：先合并再推送：
    git pull origin main --allow-unrelated-histories git push -u origin main
我的经历：因为文件是从 GitHub 下载 ZIP 得到的，本地重新 init 后历史对不上， 最后用 --force 推送成功。以后换电脑直接用 git clone，避免再走弯路。
Step 6

换电脑怎么拉代码？

新电脑上不要下载 ZIP，用 clone 一次到位：

git config --global user.name "你的名字" git config --global user.email "你的邮箱" cd ~/Documents git clone https://github.com/QIII01/QIII01.github.io.git cd QIII01.github.io

clone 完成后你自动拥有：

  • 完整文件和 Git 历史
  • 已配置好的 origin 远程
  • 本地 main 跟踪 origin/main

改完代码后：

git add . git commit -m "描述改了什么" git push
Step 7

日常更新流程

网站改完后，固定三步：

git add index.html # 或 git add . git commit -m "update homepage" git push

常用检查命令：

git status # 看哪些文件改了、有没有 staged git log --oneline # 看提交历史 git branch -vv # 看本地分支是否跟踪远程
  • working tree clean：没有未提交的改动
  • up to date with 'origin/main'：本地和 GitHub 已同步
Bonus

ZIP 下载和 git clone 的区别

git clone 下载 ZIP
带 Git 历史 ✅ 有 ❌ 没有
自带 remote ✅ 有 ❌ 要手动 git init
直接 push ✅ 可以 ❌ 容易历史冲突
适合换电脑 ✅ 推荐 ❌ 不推荐
一句话总结：想长期维护项目，永远优先 git clone； ZIP 只适合「看一眼代码」，不适合当作开发起点。
